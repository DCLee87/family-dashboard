# ADR-022: E1 기술 스택

- 상태: 승인, 실행 검증 필요
- 결정일: 2026-07-13

## 배경

DS216+II는 Intel x86 계열 CPU를 사용하지만 메모리가 1 GB이므로 여러 런타임과 데이터베이스 컨테이너를 상시 실행하는 구조는 위험하다. 동시에 PC, 삼성 TV, iOS, Android와 태블릿에 하나의 UI를 제공해야 한다.

## 결정

- 프런트엔드는 TypeScript와 React로 PWA를 작성한다.
- React는 빌드 단계에서 정적 HTML, JavaScript와 CSS로 변환하며 NAS에서 Node.js 프런트엔드 서버를 실행하지 않는다.
- 백엔드는 Go 단일 실행 파일로 HTTP API, 정적 파일 제공, 예약 작업과 상태 확인을 담당한다.
- 데이터 저장소는 SQLite를 사용하고 데이터 파일을 컨테이너 외부 NAS 볼륨에 저장한다.
- 실시간 화면 갱신은 SSE를 우선 검증하고 지원이 불안정한 기기에는 HTTP polling을 제공한다.
- 컨테이너 이미지는 다단계 빌드를 사용해 Node.js, Go 컴파일러와 빌드 캐시를 런타임 이미지에서 제외한다.
- E1은 인증·전체 도메인 기능보다 최소 화면, 상태 확인, SQLite 영속성, 재시작과 자원 사용량을 먼저 검증한다.

## 대안

### 전체 TypeScript + Node.js

- 장점: 하나의 언어와 빠른 초기 개발
- 단점: DS216+II 런타임 메모리 위험이 상대적으로 큼

### Python

- 장점: 구현과 라이브러리 사용이 쉬움
- 단점: 런타임과 패키지 크기, 메모리 예산에서 Go보다 불리할 가능성

### Rust

- 장점: 낮은 런타임 자원과 강한 안전성
- 단점: 초기 구현 복잡도와 개발 속도 부담

## 결과

- NAS 런타임은 Go 프로세스 하나와 SQLite 파일을 중심으로 단순화된다.
- 프런트엔드와 백엔드는 빌드 단계가 분리되지만 하나의 컨테이너 이미지로 배포된다.
- Go SQLite 드라이버, React 빌드 도구와 컨테이너 기반 이미지는 E1 구현 시 버전을 고정하고 공급망 검토를 해야 한다.
- 이 선택은 DS216+II 실측에서 메모리·안정성 목표를 충족한 뒤 Elaboration 아키텍처 기준선으로 확정한다.
